Project Overview</strong></h1><p>This document is the complete build specification for the redesigned IRCTC railway booking app — starting with the Login and Authentication module. It is written for use in Cursor (AI code editor) or Antigravity (design-to-code tool) and contains all research findings, design decisions, component specifications, screen anatomy, interaction states, colour tokens, and the step-by-step build roadmap.</p><h2><strong>1.1 Why this project</strong></h2><p>IRCTC is the only official channel for booking Indian Railway tickets. With over 8 million daily transactions, it is one of the highest-traffic booking systems in the world — yet it consistently scores below 2 stars on consumer review platforms. Users face CAPTCHA on every login, sessions that expire mid-payment, jargon that non-technical users cannot understand, and a UI designed for desktop that was awkwardly ported to mobile.</p><p>RailOne (launched July 2025) is the Indian Railway's newer unified platform. It solves some of these problems — mPIN login, biometric auth, no CAPTCHA, SSO across services — but still lacks persistent sessions during UPI app switches, Tatkal pre-auth, and a modern visual language.</p><p>This redesign draws the best from both platforms, adds missing features, and wraps everything in a Neumorphic design language that is tactile, trustworthy, and optimised for India's connectivity and hardware conditions.</p><h2><strong>1.2 Key research numbers</strong></h2><table><tr><td><p><strong>1.7 / 5 stars</strong></p></td><td><p>IRCTC average rating across PissedConsumer (314 reviews) — users mostly dissatisfied</p></td></tr><tr><td><p><strong>7 in 10</strong></p></td><td><p>Tatkal users report all seats vanish within the first minute of booking opening</p></td></tr><tr><td><p><strong>66%</strong></p></td><td><p>Of outage complaints are website/server related vs 21% app-specific</p></td></tr><tr><td><p><strong>32%</strong></p></td><td><p>Of users resort to travel agents just to secure a Tatkal ticket</p></td></tr><tr><td><p><strong>32 features</strong></p></td><td><p>Combined login-ecosystem features across IRCTC + RailOne — all incorporated in this spec</p></td></tr></table><h1><strong>2. Design Principles</strong></h1><p>Every decision in this build must trace back to one of these five principles. Use them to resolve any ambiguity during development.</p><h2><strong>Principle 01 — Speed over comprehensiveness</strong></h2><ul><li>The primary job is booking a ticket as fast as possible</li><li>Every feature not in the critical path goes in 'advanced' settings — not on the main screen</li><li>A user should be able to search, select and confirm in under 90 seconds</li></ul><p><strong>→  </strong><em>Remove before you add. If a feature slows the booking flow, it does not belong there.</em></p><h2><strong>Principle 02 — Trust at every step</strong></h2><ul><li>India's railway users have been burned by payment failures, phantom deductions, opaque refunds</li><li>The redesign must signal trustworthiness — confirmation states, refund timelines, transparent errors</li><li>Every error must tell the user exactly what happened and what to do next</li></ul><p><strong>→  </strong><em>No vague error codes. Every error has a plain-language explanation and a recovery action.</em></p><h2><strong>Principle 03 — Speak plain Indian</strong></h2><ul><li>Replace GNWL, PQWL, RAC, SL, 2A, 3A with human language everywhere in the UI</li><li>'Sleeper class — no AC' is clearer than 'SL'. '50% chance of confirmation' beats 'WL#23'</li><li>Support Hindi as a first-class language, not an afterthought — on the login screen from day one</li></ul><p><strong>→  </strong><em>If a new user cannot understand it without Googling, rewrite it.</em></p><h2><strong>Principle 04 — Designed for India's network</strong></h2><ul><li>Millions of users are on 3G or patchy connections, outdoors in sunlight, on budget Android phones</li><li>Every screen must work offline for core functions (cached login, offline ticket access)</li><li>High contrast, large tap targets, no animations required to communicate state</li></ul><p><strong>→  </strong><em>Skeleton screens, offline ticket access, and low-data mode are requirements — not extras.</em></p><h2><strong>Principle 05 — Reward loyalty silently</strong></h2><ul><li>Frequent travellers get the app to learn their preferences — no loyalty programme UI needed</li><li>Smart defaults: preferred class, saved routes, saved passengers — they just work</li></ul><p><strong>→  </strong><em>Smart defaults beat explicit settings. The app adapts, not the user.</em></p><table><tr><td><p><em>The Taxi Test (North Star): A person in a moving auto-rickshaw, on patchy 3G, in bright sunlight, booking a Tatkal ticket with one hand — should complete the full flow in under 2 minutes. Neumorphic Design System</strong></h1><p>Neumorphism (soft UI) creates elements that appear to extrude from or press into the background surface — made from the exact same 'material'. Two shadows create the illusion: a dark shadow bottom-right, a white/light shadow top-left. Every component in the login module uses this system.</p><h2><strong>6.1 The Core Technique</strong></h2><p>The key principle: background, card, and elements all share the same base colour. Shadow offsets make elements appear to float above or sink below the surface.</p><p>/* Raised element (buttons, cards, icon buttons) */</p><p>box-shadow: 6px 6px 12px #B8BEC7, -6px -6px 12px #FFFFFF;</p><p>/* Sunken element (text inputs — pressed into surface) */</p><p>box-shadow: inset 3px 3px 7px #B8BEC7, inset -3px -3px 7px #FFFFFF;</p><p>/* Active/pressed state (button being tapped) */</p><p>box-shadow: inset 2px 2px 5px #B8BEC7, inset -2px -2px 5px #FFFFFF;</p><p>/* Flat/disabled state */</p><p>box-shadow: none; opacity: 0.5;</p><table><tr><td><p><em>Critical constraint: The background cannot be pure white (#FFFFFF) or pure black (#000000).
